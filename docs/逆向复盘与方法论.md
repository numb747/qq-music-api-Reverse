# QQ 音乐 Web 加密逆向 · 完整复盘与方法论

> 本文记录 y.qq.com 请求签名(`sign`)与请求体加密(AES-GCM)的完整逆向过程、
> 所用工具、踩过的坑、以及可迁移的方法论。既是项目复盘,也是一份 JS VMP 逆向的实战笔记。

---

## 一、这个项目是什么,我如何评价它

### 1.1 项目本质

QQ 音乐网页版所有核心业务(搜索、歌单、歌词、播放地址)都收敛到一个统一入口
`u.y.qq.com/cgi-bin/musicu.fcg`。当请求需要鉴权时,前端会:

1. 对明文请求体做 **SHA-1 变换**生成 `sign` 签名;
2. 用 **AES-GCM** 加密请求体,端点切换到 `musics.fcg`,附加 `encoding=ag-1`;
3. 服务端返回的响应同样是 AES-GCM 密文,前端再解密。

这三段逻辑(签名、加密、解密)全部封装在 **VMP(虚拟机保护)** 里 —— 把原始运算
拆散成字节码,由一个 `switch(opcode)` 解释器逐条执行,以此对抗静态阅读。

本项目做的事:**把 VMP 引擎连同签名/加解密函数从浏览器里抽出来,在 Node 中
独立还原了完整的 加密 → 签名 → 发送 → 解密 链路**,最终形成一个通用客户端
`callQQ(module, method, param)`,可脱离浏览器调用任意接口。

### 1.2 我的评价

**做得好的地方:**

- **路径选择务实**。没有陷入"必须把 VMP 每条指令都翻译成人类可读代码"的执念,
  而是抓住"VMP 保护的是过程,但函数入口是暴露的"这个要害,直接取函数引用。
  这是整个项目最关键的判断,省下了可能数天的字节码翻译工作。
- **双工具协同**。Burp 负责流量层(看结果、重放、抓包),DevTools 负责代码层
  (断点、取函数),各司其职,互相印证。
- **实证驱动**。每一步结论都用实验验证,而不是猜测 —— 比如"sign 末段不参与
  校验"是靠改一位重放试出来的,不是读代码读出来的。
- **闭环完整**。从"能不能看到流量"一路做到"独立脚本批量拿数据",没有半途而废。
